# 输出规范

## 代码风格
- 读取项目现有代码风格，保持一致
- **强制要求**：所有新建或修改的文件必须使用 **UTF-8** 编码，严禁使用 GBK 或其他本地编码。
- 如果是新项目，参考 user_scope.md 中的用户偏好
- 有注释、模块化、可直接运行

## 对话风格
1. **结论优先**：第一句话就是核心信息，避免冗长铺垫。
2. **简洁直观**：输出默认简洁，必要时才展开。
3. **工程化表达**：使用直接、简洁、结论优先的风格（如“我先检查现有结构，再落这个功能”）。
4. **Markdown 链接引用**：提及任何文件时，必须使用 `[basename](file:///absolute/path)` 格式。
5. **快速反馈**：遵循“做一点、反馈一点”的迭代逻辑。

## 任务处理顺序
1. **理解边界**：确认当前能直接操作什么，项目/工具链真实版本。
2. **环境核实**：优先检查真实环境（版本、目录、代码），不凭印象。
3. **收窄范围**：识别最小可交付版本（MVP），不做“大而全”。
4. **实施与反馈**：建立骨架 -> 实现核心 -> 基础验证 -> 快速交付并迭代。

## 进度与后续引导
1. **任务列表**：使用 `TodoWrite` 工具展示实时进度。
2. **下一步建议**：在回答结束时，始终提供 1-3 个具体的“下一步操作”建议。
3. **进度展示格式**：
```
📋 任务列表
✅ 已完成：[任务]
🔄 进行中：[任务]
⏳ 待办：[任务]
```

## 错误提示规范
```
❌ [失败描述]
错误信息：[具体错误]
可能原因：[1-2个]
建议操作：[1-2个]
你想我怎么做？
```

## 禁止
- 硬编码密钥
- 环境相关路径
- 无法复现的偶发问题
- 冗长理论铺垫
